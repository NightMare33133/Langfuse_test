# Langfuse RAG 评测工具需求文档

## 1. 背景

我现在在使用 `Dify + Langfuse` 测试一个 RAG 工作流。

当前现状：

- Dify 的运行过程已经成功接入 Langfuse
- 我可以从 Langfuse 导出 tracing 数据，格式是 `JSONL`
- 我已经有一份 Python 代码，能够把原始 Langfuse 导出重组为“每次问答一条样本”
- 但是每次手动运行脚本、传命令行参数、翻 JSON 看结果都比较麻烦

我希望做一个**本地轻量评测工具**，把“导入 tracing -> 整理样本 -> 用 LLM 自动评判 -> 可视化查看结果”这一整套流程串起来。

这个工具是为了做 **RAG 评测**，不是为了重新做聊天机器人本体。

## 2. 目标

构建一个本地小工具，帮助我把 Langfuse 导出的 tracing 数据转成结构化 RAG 评测结果，并尽量减少手动操作。

核心流程如下：

1. 导入 Langfuse 导出的 `JSONL`
2. 按 `trace_id` 重组事件
3. 提取每次问答中的关键字段
4. 保存为结构化样本
5. 将样本发送给一个 judge LLM 进行评判
6. 输出评测结果
7. 以页面方式查看每条样本和整体指标

## 3. 目标用户

主要用户：

- 我自己

用户特征：

- 会用 Python
- 不想频繁手动跑命令行脚本
- 想要一个更直观、更可视化的本地工具

## 4. MVP 范围

### 4.1 导入原始数据

工具应支持：

- 从本地选择 Langfuse 导出的 `JSONL` 文件
- 显示基础信息：
  - 文件名
  - 文件大小
  - 总行数

### 4.2 解析 Langfuse 导出

工具需要将 Langfuse 原始 event 级别日志，按 `trace_id` 重组为 trace 级别样本。

每条样本至少提取这些字段：

- `trace_id`
- `question`
- `retrieval_query`
- `retrieval_results`
- `llm_input`
- `llm_output`
- `final_answer`

可选元数据：

- `session_id`
- `user_id`
- `workflow_run_id`

处理完成后，需要保存为：

- `processed/langfuse_samples.jsonl`
- `processed/langfuse_summary.json`

### 4.3 Trace 查看页面

工具应提供一个页面/表格查看解析后的样本。

每行建议展示：

- question
- final answer 摘要
- retrieval results 数量
- trace id

点击某一行后，打开详情区域，展示：

- 原始问题
- retrieval query
- 检索结果列表（按 position 排序）
- LLM input
- LLM output
- final answer

### 4.4 Judge LLM 评判

工具应支持将解析后的样本发送给一个 judge LLM。

judge 输入至少包括：

- question
- retrieval results
- final answer

如果后续补充 gold answer 或标准证据，也应支持一并传入。

judge 输出应为结构化 JSON，字段例如：

- `retrieval_top1_hit`
- `retrieval_top3_hit`
- `retrieval_top5_hit`
- `answer_correct`
- `reason`

第一版先使用二值评分：

- `0` / `1`

### 4.5 保存评测结果

工具应将评测结果保存为：

- `judged/eval_results.jsonl`

每条结果至少包括：

- `trace_id`
- `question`
- judge scores
- `reason`

### 4.6 指标看板

工具应展示整体评测指标，例如：

- total traces
- evaluated traces
- top1 hit rate
- top3 hit rate
- top5 hit rate
- answer correctness rate

并支持筛选：

- 全部样本
- retrieval 失败样本
- final answer 错误样本

## 5. 非目标

MVP 第一版不需要：

- 直接通过 Langfuse API 拉数据
- 把 scores 回写到 Langfuse
- 多用户登录
- 云部署
- 复杂权限系统
- 很高级的图表
- 自动构造 gold dataset

这些都可以后续再加。

## 6. 推荐交互方式

工具整体风格应尽量：

- 本地优先
- 轻量
- 操作简单

推荐流程：

1. 首页：导入文件
2. 解析页面：执行 parsing
3. 样本查看页面：浏览 trace 详情
4. 评测页面：调用 judge LLM
5. 结果页面：查看整体指标和失败案例

推荐布局：

- 左侧导航栏
- 中间主表格
- 右侧详情面板或弹窗

## 7. 文件结构建议

建议项目结构如下：

```text
Langfuse_test/
├─ main.py
├─ data/
│  ├─ raw/
│  │  └─ *.jsonl
│  ├─ processed/
│  │  ├─ langfuse_samples.jsonl
│  │  └─ langfuse_summary.json
│  └─ judged/
│     └─ eval_results.jsonl
├─ prompts/
│  └─ judge_prompt.txt
└─ app/
   └─ ...
```

## 8. 技术偏好

优先技术栈：

- Python
- 本地运行

推荐选项：

- `Streamlit`
- 或者 `Gradio`

原因：

- 文件上传和表格展示简单
- 本地快速起页面
- 很适合做这种“数据处理 + 结果查看”的内部工具

如果可能，优先复用 `main.py` 中现有的 parsing 逻辑，不要从零重写。

## 9. 功能需求

### FR-1 文件导入

系统必须允许用户从本地选择 Langfuse 导出的 `JSONL` 文件。

### FR-2 日志解析

系统必须将 event 级别 Langfuse 日志重组为 trace 级别样本。

### FR-3 结果落盘

系统必须将处理后的结果保存到磁盘。

### FR-4 样本查看

系统必须支持逐条查看样本内容。

### FR-5 LLM 评测

系统必须支持将样本发送给一个可配置的 judge LLM 并获取结构化评分。

### FR-6 指标汇总

系统必须根据 judge 结果计算整体指标。

### FR-7 失败案例筛选

系统应支持查看评测失败或可疑的样本。

## 10. 后续增强方向

后续可考虑：

- 直接接 Langfuse API，而不是手动导出文件
- 将评分结果写回 Langfuse
- 导出 Markdown 报告
- 对比两个实验版本
- 手动录入 gold answer
- 在 UI 中编辑 judge prompt
- 支持只重跑某些样本

## 11. 验收标准

如果满足以下条件，则 MVP 视为完成：

1. 我可以导入 Langfuse `JSONL`
2. 我可以把 raw tracing 解析成 trace 级别样本
3. 我可以在页面中查看每一条样本
4. 我可以让 judge LLM 自动评判样本
5. 我可以看到整体指标：
   - top1
   - top3
   - top5
   - answer correctness
6. 我可以方便地查看失败案例，而不是手动翻原始 JSON

## 12. 推荐实现顺序

Phase 1：

- 把现有 `main.py` 的 parsing 能力封装成可复用函数
- 用 Streamlit 做一个最基础的文件导入 + 解析页面

Phase 2：

- 增加样本表格和详情查看页

Phase 3：

- 加入 judge LLM 调用
- 保存 `eval_results.jsonl`

Phase 4：

- 增加指标面板
- 增加失败案例筛选

## 13. 额外说明

- Langfuse 原始导出中，`input`、`output`、`metadata` 等字段可能是“JSON 字符串套 JSON”
- 解析逻辑需要容忍字段缺失
- retrieval results 内容可能非常长，UI 中应默认折叠/截断，并支持展开查看
- 第一版重点是“稳定 + 清晰 + 好用”，不必优先追求很复杂的设计
